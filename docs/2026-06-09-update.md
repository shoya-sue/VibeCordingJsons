## 品質分析サマリー

### 権限プロンプト問題の根本原因（ユーザー報告）

ユーザー指摘: 「なぜ毎度 `curl https://api.github.com/...compare...` 程度で許可を求められるのか。人間の仕分けに限界、改善して」。

- 直接原因: `update-release` skill step 0-a が「fetch from `https://github.com/...`」とだけ書いており、ツールを指定していない。サブエージェントが `curl` を選び、`Bash(curl *)` (ask) に当たる。
- `Bash(gh *)` と `WebFetch(*)` は **既に allowlist 済み**。`gh api repos/.../compare/...` を使っていればプロンプトは出なかった。
- 「read-only curl を allow に追加」は **プラットフォーム仕様上不可**（公式ドキュメントで確定）:
  - 評価順は `deny → ask → allow`（first match wins）。`curl` allow ルールは既存 `Bash(curl *)` ask に shadow され、依然プロンプトする。
  - Bash 引数マッチは脆弱（公式の警告枠が `Bash(curl http://github.com/ *)` の例で「フラグ前置・リダイレクト・余分な空白で外れる」と明記）。
  - 公式推奨は「curl/wget を deny し、`WebFetch(domain:...)` を allow」。
- ユーザー選択: **ワークフロー修正のみ**（curl を deny に落とす公式ハーデニングは見送り、手動 curl の escape hatch を残す）。
