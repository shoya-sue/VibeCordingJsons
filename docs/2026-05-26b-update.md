## 設計判断

### awk vs python3

過去 v0.44.2 で **BSD awk の newline-in-`-v` failure** に遭遇し python3 へ
切替えた経緯がある。今回 awk を採用したのは:

- frontmatter の構造解析を要する **inline 処理** であり、外部マーカーなし
- `-v` を使わず inline pattern のみ
- 1 ファイル当たりの処理量が小さく、Python interpreter 起動コストの方が
  累積するとボトルネック化する可能性

検証として、`bash strip-memory-provenance.sh` を 2 回連続実行して idempotent
であることを確認 (2 回目は 0 行 strip)。

### 3 ディレクトリ走査

- `$HOME/.claude/projects/-Users-shoya-sue-Public/memory/` — auto-memory source
  (symlinked to vault)
- `$VAULT/30_knowledge/claude-code/memory/Public/` — manual mirror
- `$VAULT/90_artifacts/claude-code/memory/**/*.md` — legacy sync target

source と mirror は symlink 経由で同一 inode のことが多いが、 race を避けるため
両方走査。`find -type f` は symlink を follow するので問題なし。

### Stop chain 順序

```
1. ECC plugin session-end.js   (10s)
2. obsidian-session-end.sh     (15s)
3. obsidian-auto-capture.sh    (5s)
4. strip-memory-provenance.sh  (10s)  ← 新規
```

auto-capture の後に置いた理由: auto-capture が memory ファイルに書き込む可能
性は無いが、 安全側に倒して capture 完了後に strip。
